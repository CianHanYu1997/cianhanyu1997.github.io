IPv6 逐一嘗試導致 Google API 超時的完整排查過程

背景說明

用戶透過信箱登入系統時，從填完表單、按下登入，到看到「請收驗證信」畫面，整個流程需要等待約 25 秒。

系統環境：Django 後端、Nginx + Gunicorn、部署於 AWS EC2、登入流程使用 Magic Link（信箱驗證碼）、reCAPTCHA 用於人機驗證。

排查步驟

Step 1：Nginx Access Log — 確認延遲區間

先從最外層的 Nginx access log 確認延遲發生在哪個 HTTP 環節。

從使用者回報與 access log 時間點來看，延遲集中在登入流程的 POST 階段；但 access log 本身還看不出單一 request 的實際處理時間，所以接下來需要補上 request timing 來確認。

Step 2：加入 Nginx Request Timing — 定位在後端

Nginx 預設 access log 不含耗時資訊。修改 log_format，補上兩個關鍵欄位：

  • rt（request_time）：Nginx 從收到請求到回應完成的總時間
  • urt（upstream_response_time）：Nginx 等後端（Gunicorn）回應的時間

兩個數值幾乎相同，代表延遲完全來自後端 Django，Nginx 本身幾乎沒有處理時間。

Step 3：View 層埋細粒度 Log — 定位到 reCAPTCHA

在 Django 的登入 view 中，逐一記錄每個步驟的耗時（使用 time.monotonic() 計算 elapsed 與 total）：

24.121 秒全部消耗在 recaptcha_request，後面的 DB 查詢、建立 magic link、發送 Email 加起來不到 0.3 秒。

整體登入流程與延遲發生點如下：

sequenceDiagram
    participant U as User
    participant N as Nginx
    participant D as Django
    participant G as Google reCAPTCHA
    participant S as SMTP

    U->>N: POST /auth/login
    N->>D: 轉發請求
    D->>G: requests.post(siteverify, timeout=3)

    Note over D,G: DNS 解析出多個 IPv6 位址
    Note over D,G: 逐一 connect，每個位址最多等約 3 秒
    Note over D,G: 多次 IPv6 timeout 累積約 24 秒

    D->>G: 最後 fallback 到 IPv4
    G-->>D: 200 OK

    D->>D: user_lookup / create_magiclink
    D->>S: send mail
    S-->>D: 郵件送出成功

    D-->>N: 200 OK
    N-->>U: 顯示「請收驗證信」

注意：為什麼有設定 timeout=3 卻還是有 24 秒延遲？

requests 的 timeout 不是整個 request 的總時限，而是每次 socket connect/read 的時限。

DNS 解析 www.google.com 時會回傳多個 IPv6 位址，requests/urllib3 會逐一嘗試每個位址。每個 IPv6 connect 最多等 3 秒，約 8 個位址 × 3 秒 ≈ 24 秒累積，全部失敗後才 fallback 到 IPv4 成功。

所以最終 status=200，但總耗時已被拖長。timeout=3 其實有在每次 connect 上生效，只是它被觸發了 8 次。

Step 4：curl 驗證 IPv4 / IPv6 — 確認根因

現代 Linux 系統會依照 RFC 6724 的位址選擇規則排序連線目標；在這台機器上，DNS 同時解析到 IPv4 和 IPv6 位址時，會優先嘗試 IPv6。因此分別測試兩種協定對 Google reCAPTCHA API 的連線能力：

結論：

  • IPv4 走 0.1 秒即得到 HTTP 200 正常回應
  • DNS 回傳多個 IPv6 位址，每個 connect 最多等 3 秒（timeout=3），共嘗試約 8 個位址
  • 累積約 8 × 3 = 24 秒後全部失敗，才 fallback 到 IPv4 成功

根本原因

當時這台 EC2 所在的網路環境對 IPv6 外連 Google API 不通，但 OS 的位址選擇策略讓 IPv6 具有更高優先權。每次呼叫 reCAPTCHA API，requests/urllib3 遵從 OS 的位址選擇策略，逐一嘗試所有 IPv6 位址，每個都等到 connect timeout，累積約 24 秒後才 fallback 到 IPv4 成功。

修正方式

在 /etc/gai.conf 調整位址選擇優先權，讓 IPv4-mapped 位址（::ffff:0:0/96）優先於純 IPv6：

修正後，reCAPTCHA 呼叫改走 IPv4，耗時從 24 秒降至約 0.4 秒，整體登入流程回歸正常。

什麼是 gai.conf？

/etc/gai.conf 是 Linux 的 getaddrinfo() 設定檔，控制 DNS 解析出多個位址時的優先順序。

precedence ::ffff:0:0/96 100 代表：當遇到 IPv4-mapped IPv6 位址時，給予優先權值 100（數字越大越優先）。這讓 glibc 在同時有 IPv4 和 IPv6 選項時優先選擇 IPv4。

這是系統層級的位址選擇設定，會影響同一台機器上使用 glibc getaddrinfo() 的程序；它不會停用 IPv6，但會改變本機地址排序，所以正式調整前要確認同機其他服務是否依賴 IPv6 優先連線。

排查方法論整理

這次排查的核心思路是「由外而內、逐層縮小範圍」：

層次 工具 得到的資訊
HTTP 層 Nginx access log 哪個 request 慢
網路層 Nginx rt / urt 問題在 Nginx 還是後端
應用層 View 埋點 log 哪個步驟卡住
協定層 curl -4 / -6 網路環境對 IPv4/IPv6 的支援差異

每一步都有明確的「假設 → 驗證」動作，而不是直接猜測根因：

  • rt ≈ urt → 問題在後端，不在 Nginx
  • 所有 view 步驟 < 0.3s，總計 > 24s → 問題在外部 API 呼叫
  • 外部 API 耗時 24s 但最終 200 → 問題在連線協定，不在 API 本身
