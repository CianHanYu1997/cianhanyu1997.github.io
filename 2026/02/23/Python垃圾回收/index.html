Python的垃圾回收

Cinahan Yu Lv1
  2026-02-23 07:25:18 2026-02-23 07:25:18 撰寫   2026-04-16 15:46:02 2026-04-16 15:46:02 更新

Python 透過引用計數、標記清除與分代回收三種機制自動管理記憶體，讓開發者無需手動釋放記憶體。
了解這些機制，才能寫出更高效、不易發生記憶體洩漏的程式。

本文概要：

  • 什麼是垃圾回收？
  • 三種回收算法：引用計數、標記清除、分代回收
  • 垃圾回收機制的潛在隱憂

1. 什麼是垃圾回收？

前篇 Python的語言特性 提到「內存地址」，每次創建一個物件時，Python 都會分配一塊記憶體位址給它。
由於記憶體空間有限，垃圾回收機制（Garbage Collection） 就負責自動釋放不再使用的記憶體。

  • 底層語言（如 C）：開發者必須手動管理記憶體。
  • 高階語言（如 Python）：自動處理，讓開發者專注於商業邏輯。

若完全不了解垃圾回收機制，寫出來的程式往往容易有效能瓶頸！

2. 垃圾回收算法

Python 主要以引用計數為核心，搭配標記清除與分代回收作為輔助。

2-1 引用計數（Reference Counting）

每個物件都有一個「引用計數器」：

  • 每當物件被引用一次，計數器 +1
  • 引用被刪除時，計數器 -1
  • 計數器歸零時，該記憶體位址立即釋放

getrefcount 本身也會產生一次臨時引用，所以結果為 2。

優缺點

說明
優點 簡單高效（只需對計數器做加減），物件一旦計數歸零即立即釋放記憶體
缺點 無法解決循環引用（A 引用 B、B 引用 A，兩者永遠無法回收）；物件嵌套時需層層檢查，耗費較多時間

2-2 標記清除（Mark and Sweep）

專門用來解決引用計數無法處理的循環引用問題，分為兩個步驟：

  1. 標記階段：從 GC Root 出發，追蹤並標記所有可達物件（Reachable Object）。
  2. 清除階段：回收所有未被標記的物件所占用的記憶體。

缺點：會觸發 Stop-the-World（STW），標記清除期間整個應用程式將完全暫停。

名詞說明

  • GC Root：程式的起點，如全域變數，是追蹤引用鏈的起始節點。
  • Reachable Object：能從 GC Root 沿引用鏈找到的物件，即「仍在使用中」的物件。

3. 分代回收（Generational Collection）

為了降低 STW 造成的程式卡頓，Python 引入分代回收，將物件依「存活時間」分為三個世代：

世代 說明 回收頻率
G0（第零代） 剛建立的物件 最頻繁
G1（第一代） 在 G0 清理中存活，晉升至此 中等
G2（第二代） 存活最久的物件 最少

大多數物件「生命週期很短」，因此集中清理 G0 即可回收大量記憶體，大幅減少對 G1、G2 的掃描次數。

4. 垃圾回收的潛在隱憂

  1. 效能開銷：GC 觸發的時間與耗時難以預測，可能造成程式突然停頓數毫秒甚至數秒。對於「極低延遲」系統（如自動駕駛、高頻交易、即時遊戲）而言，這可能是致命傷。

  2. 記憶體碎片化：GC 清理後，記憶體中會留下不連續的空隙，導致記憶體分配速度隨時間逐漸下降。

小提示：如果需要進行大量純數值運算，建議改用 Pandas 或 NumPy。
它們在記憶體中採用連續存放，沒有碎片化問題，效能也更為優異。
